title: Quickstart
description: "Install zombiectl, run /usezombie-install-platform-ops, see a real diagnosis in Slack."
---

This walks through installing the flagship `platform-ops` zombie on one of your repositories. Total time: about ten minutes from a cold machine. At the end, a deploy failure on the repo lands an evidenced diagnosis in your Slack channel.

<Steps>

<Step title="Prerequisite: a host agent">
  You'll drive the install via a markdown-skill host. Any of these work — the install skill ships identical prompts in each:

  - **Claude Code** — `claude.ai/code`
  - **Amp** — `ampcode.com`
  - **Codex CLI** — `npm install -g @openai/codex`
  - **OpenCode** — `opencode.ai`

  Pick one. The rest of this page assumes you have a working session in front of you.
</Step>

<Step title="Install zombiectl">
  ```bash
  npm install -g @usezombie/zombiectl
  ```

  Verify:

  ```bash
  zombiectl --version
  ```

  `zombiectl` is the only binary you install on your machine. It ships the `/usezombie-*` skill samples under `~/.config/usezombie/samples/` so the install skill (next step) can drive `zombiectl install --from` against a known-good source.
</Step>

<Step title="Sign in">
  ```bash
  zombiectl login
  ```

  This opens a Clerk OAuth flow in your browser. New accounts come with a `$5` starter credit pool — never expires, covers both hosted execution and inference against the platform-managed default model.
</Step>

<Step title="Create a workspace">
  ```bash
  zombiectl workspace add platform-ops
  ```

  Workspaces are the boundary for credentials, access control, and webhook namespaces — every zombie lives in exactly one. The command creates a new workspace under your tenant, returns its `workspace_id`, and sets it active for subsequent commands. You can switch later with `zombiectl workspace use <id>`. See [Command reference → workspaces](/cli/zombiectl#workspaces).
</Step>

<Step title="Install and run the install skill">
  The `usezombie-install-platform-ops` skill is served as a single host-neutral SKILL.md from `https://usezombie.sh/skills.md`. Drop it into your host agent's skills folder:

  ```bash
  # Claude Code — adjust the destination for Amp / Codex CLI / OpenCode (their per-host skills paths differ)
  mkdir -p ~/.claude/skills/usezombie-install-platform-ops
  curl -fsSL https://usezombie.sh/skills.md \
    -o ~/.claude/skills/usezombie-install-platform-ops/SKILL.md
  ```

  Then, in your host agent, invoke:

  ```
  /usezombie-install-platform-ops
  ```

  The skill drives `zombiectl install --from ~/.config/usezombie/samples/platform-ops` under the hood. Power users can run that directly; everyone else gets the guided flow.

  The skill asks three gating questions:

  - **Slack channel** — where diagnoses post (e.g. `#platform-ops`).
  - **Production branch glob** — which branches count as production (default `main`).
  - **Cron schedule** (optional) — for periodic health checks (e.g. `*/30 * * * *`). Leave blank for webhook-only.

  It writes `.usezombie/platform-ops/SKILL.md` and `.usezombie/platform-ops/TRIGGER.md` into the current repo. Re-running against the same workspace is idempotent; existing files are diffed and updated.
</Step>

<Step title="Wire the GitHub webhook">
  At the end of the install flow, the skill prints two values:

  - **Webhook URL** — `https://api.usezombie.com/v1/webhooks/{zombie_id}` (your `zombie_id` is shown in the install output, e.g. `zmb_2041`).
  - **Webhook secret** — a one-time random value, shown **once**. Copy it now.

  In your GitHub repo: **Settings → Webhooks → Add webhook**.

  - **Payload URL** — paste the webhook URL.
  - **Content type** — `application/json`.
  - **Secret** — paste the webhook secret.
  - **Events** — select **Workflow runs** (the GitHub UI label; only the `workflow_run` event is needed). The zombie further filters incoming payloads for `conclusion=failure` on a production-branch workflow — the rule lives in the generated `.usezombie/platform-ops/SKILL.md` and is editable. See [Authoring](/zombies/authoring) for how to widen or narrow it.

  <Tip>
    **One workspace credential, many repos.** The same webhook secret works for every zombie in this workspace — paste it into N repo webhook configs. Tradeoff: rotation is workspace-wide; rotating the workspace `github` credential affects every zombie that uses it.
  </Tip>

  <Warning>
    **Lost the secret?** It's not stored anywhere user-visible after install — you have to rotate. Two steps:

    **1. Rotate the workspace `github` credential.** Generate a new webhook secret (e.g. `openssl rand -hex 32`), store it in 1Password, then pipe the credential JSON through `op inject` so the secret never appears in shell history or argv. Run `op signin` first — if the 1Password session is stale, `op inject` fails before `zombiectl` runs and the workspace vault is left untouched (don't proceed to step 2 in that case, or your GitHub webhook will be set to a secret the server never stored):

    ```bash
    cat <<'JSON' | op inject | zombiectl credential add github --data=@- --force
    {
      "api_token":      "{{ op://Personal/usezombie-github/api_token }}",
      "webhook_secret": "{{ op://Personal/usezombie-github/webhook_secret }}"
    }
    JSON
    ```

    `op inject` substitutes each `{{ op://... }}` reference with the resolved secret and pipes the rendered JSON to `zombiectl`. `--data=@-` reads from stdin; `--force` overwrites the existing credential. If you don't use 1Password, the literal-JSON form below works the same — just keep it on stdin, never inline:

    ```bash
    cat <<'JSON' | zombiectl credential add github --data=@- --force
    { "api_token": "ghp_...", "webhook_secret": "<new-secret>" }
    JSON
    ```

    See [Credentials](/zombies/credentials) for the full schema.

    **2. Update every repo's webhook.** In each GitHub repo using this workspace, go to **Settings → Webhooks → Edit** and paste `<new-secret>` into the **Secret** field. The old secret stops working the moment step 1 completes, so do this immediately after.
  </Warning>
</Step>

<Step title="Trigger a real diagnosis">
  Cause a deploy failure on the production branch — push a known-broken commit, fail a test on purpose, whatever you have handy. Within seconds:

  - GitHub fires `workflow_run.completed` with `conclusion: failure` to the webhook URL.
  - The zombie wakes, calls the tools `TRIGGER.md` allow-lists (`http_request` against your hosting provider, `memory_store` for findings), gathers evidence from the failed workflow.
  - An evidenced diagnosis posts to your Slack channel.

  Tail it — replace `<zombie_id>` with the value the install step printed (e.g. `zmb_2041`):

  ```bash
  zombiectl logs zmb_2041
  ```

  Need to slice by source or time window? `zombiectl events zmb_2041 --actor 'webhook:*' --since 30m`. Lost track of the ID? `zombiectl list` prints every zombie in the active workspace. Or watch the same stream in Mission Control — the web dashboard at `app.usezombie.com`.
</Step>

</Steps>

## What just happened

The platform-ops zombie installed `SKILL.md` (the prose system prompt — what to investigate, how to phrase a diagnosis) and `TRIGGER.md` (which tools the model can call) into your repo. From here, behaviour iterates on prose: edit the `.usezombie/platform-ops/SKILL.md` file, push, the next trigger runs the new behaviour. No redeploys, no DAG editor.
