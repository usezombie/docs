## Add a workspace

```bash
zombiectl workspace add [name]
```

The optional positional `name` is a human label; if omitted, the server picks a default. There is no repo URL — workspaces are tenant-scoped containers for credentials, zombies, and webhook namespaces. Repos enter the picture later, when you configure per-zombie webhooks (see [Quickstart](/quickstart)).

For example:

```bash
zombiectl workspace add platform-ops
```

The command creates the workspace, returns its `workspace_id`, and persists it as the active workspace in `~/.config/zombiectl/workspaces.json`.

## List workspaces

```bash
zombiectl workspace list
```

Displays all workspaces in your account with their status, connected repository, and plan tier.

## Switch the active workspace

```bash
zombiectl workspace use <workspace-id>
```

Makes the workspace active for subsequent CLI commands — `zombiectl install --from`, `zombiectl list`, `zombiectl status`, `zombiectl events`, `zombiectl steer`, and anything else that accepts a `--workspace-id` flag defaults to this value when the flag is omitted.

The choice is persisted to `~/.config/zombiectl/workspaces.json` (`current_workspace_id` field), so it survives across shells and sessions.

Errors out if the id isn't in your local list — run `zombiectl workspace list` first to see what's available.

<Tip>
  Mission Control at [app.usezombie.com](https://app.usezombie.com) has its own workspace switcher backed by a cookie. The CLI config and the Mission Control cookie are independent — switching one does **not** affect the other. Useful when you want the CLI pointed at staging while the browser watches production.
</Tip>

## Show the active workspace

```bash
zombiectl workspace show
```

Prints the active workspace's id, repo URL, default branch, and active marker. Pass `--workspace-id <id>` to inspect a specific workspace without changing the active one. Add `--json` for scripting.

This mirrors the Mission Control **Settings** page.

## Remove a workspace

```bash
zombiectl workspace remove <workspace-id>
```

Removing a workspace:

- Stops accepting new events immediately.
- Allows in-flight stages to complete.
- Retains stage history for 30 days.
- Uninstalls the GitHub App connection for that repository.

<Warning>
  This action cannot be undone. You will need to re-add the workspace and reinstall the GitHub App to reconnect.
</Warning>
