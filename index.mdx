## A long-lived runtime for one operational outcome

You have a deploy that fails at 2am. Your senior on-call engineer is asleep. You guess at the cause from a stack trace, ship a patch, hope.

UseZombie is a durable runtime that captures the senior engineer's playbook in markdown — `SKILL.md` plus `TRIGGER.md`, no workflow DAG — and runs it against your infrastructure when the trigger fires. The flagship `platform-ops` zombie wakes on a GitHub Actions deploy failure, gathers evidence from your hosting provider, your data-plane, and the failed run, and posts an evidenced diagnosis to Slack. The same zombie is reachable via `zombiectl steer` for a manual investigation.

## How a stage runs

<Columns cols={2}>
  <Card title="One loop, many event sources" icon="bolt">
    Webhook firings, scheduled cron runs the zombie sets up itself, and `zombiectl steer` messages all flow through the same reasoning. Every event lands on `core.zombie_events` with actor provenance.
  </Card>
  <Card title="Placeholder credentials" icon="shield-halved">
    The model never sees raw secret bytes. The tool bridge substitutes at the sandbox boundary; prompt-injection attacks recover only the placeholder.
  </Card>
  <Card title="Reasons past the context cap" icon="layer-group">
    Memory checkpoints, a rolling tool-result window, and stage chunking compose so deep incidents continue past the model's working-memory limit. <a href="/concepts/context-lifecycle">Context lifecycle →</a>
  </Card>
  <Card title="Approval gating" icon="circle-check">
    Risky actions block until a human clicks Approve in the dashboard or Slack. State machine survives worker restarts.
  </Card>
  <Card title="Spend ceiling" icon="gauge-high">
    Per-day and per-month dollar caps on hosted execution. `$10` starter credit that never expires. Inference cost is yours, paid direct.
  </Card>
  <Card title="Kill switch" icon="power-off">
    `zombiectl kill` checkpoints state mid-stage; nothing lost. Resume via `zombiectl steer`.
  </Card>
</Columns>
