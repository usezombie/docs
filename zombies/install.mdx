## Overview

`zombiectl install <template>` is the first command in a zombie's lifecycle. It writes a local directory containing two editable files — `SKILL.md` and `TRIGGER.md` — scaffolded from a bundled template. Nothing is uploaded, no API calls are made, and no authentication is required. You edit the files, then `zombiectl up` takes it from there.

```bash
zombiectl install slack-bug-fixer
```

```
✓ slack-bug-fixer installed.
  Created slack-bug-fixer/
    SKILL.md   — agent instructions (edit this)
    TRIGGER.md — deployment config

Next steps:
  zombiectl up              Start the zombie
  zombiectl credential add  Add credentials (optional)
```

## Bundled templates

The `zombiectl` binary ships a small set of templates intended as MVP starting points. Each is a complete, deployable scaffold.

| Template | What it does | Skills used |
|----------|--------------|-------------|
| `platform-ops` | The flagship v2 sample. Wakes on a GitHub Actions deploy-failure webhook, gathers evidence from GH run logs + Fly + Upstash, posts an evidenced diagnosis to Slack. Read-only against GitHub, Fly, and Upstash. See [zombie templates](/zombies/templates). | `github`, `fly`, `upstash`, `slack` |
| `slack-bug-fixer` | Watches a Slack channel, identifies bug reports, opens a PR against the linked repo. | `slack`, `github` |

Run `zombiectl install` with no argument to see the current bundled set in the error output.

## File layout

The installed directory contains two files. Both are plain text; both are designed to be edited.

```
slack-bug-fixer/
├── SKILL.md      # agent instructions (natural language)
└── TRIGGER.md    # machine-readable config + deployment metadata
```

### `SKILL.md`

A markdown file with free-form agent instructions. Write it as if you were onboarding a smart colleague to the job: who they are, what events they will receive, what to do with each, what tone to use, what to avoid.

```markdown SKILL.md
You are a bug-triage agent.

When a Slack message arrives in #bugs:
1. Read the message and any thread replies.
2. If it describes a reproducible bug, open a PR against the linked repo
   with a failing test and a proposed fix.
3. Reply in-thread with the PR URL and a one-line summary.
4. If it isn't a bug (question, feature request, noise), tag the thread
   with `:not-a-bug:` and stop.

Never close or merge PRs. Never speak outside the thread.
```

### `TRIGGER.md`

A markdown file with YAML frontmatter that configures how the zombie receives events, which skills it is allowed to invoke, what credentials it expects, and its budget.

```markdown TRIGGER.md
---
name: slack-bug-fixer
trigger:
  type: webhook
  source: slack
  event: message.posted
skills:
  - slack
  - github
credentials:
  - slack_bot_token
  - github_pat
budget:
  daily_dollars: 5.0
  monthly_dollars: 29.0
network:
  allow:
    - slack.com
    - api.github.com
---

This zombie triages bug reports in a Slack channel.
```

The `name` field becomes the zombie's display name. The `trigger.source` field identifies the upstream provider and determines the webhook signature scheme for first-class integrations (`slack`, `github`, `linear`, `jira`, `grafana`, `clerk`, `custom`). The `credentials:` list names the vault keys the zombie will read at event time — every entry must exist in the workspace vault before `zombiectl up` succeeds. The `network.allow:` list is an outbound hostname allowlist enforced by the credential firewall.

See [Writing skills](/zombies/skills) for the full `TRIGGER.md` reference, and [Templates](/zombies/templates) for the anatomy of a bundled template.

## Flags and options

`zombiectl install` takes a single positional argument.

| Argument / flag | Required | Purpose |
|-----------------|----------|---------|
| `<template>` | Yes | Name of a bundled template. |
| `--json` | No | Emit JSON instead of the human-readable greeting. |

Exit codes:

- `0` — template installed.
- `2` — unknown or missing template name.
- `1` — filesystem error (permission, disk full).

## After install

You have two reasonable next moves:

1. **Edit the files**, then `zombiectl up` to deploy. The bundled templates declare the vault keys they expect under `credentials:` — the platform reads them from the workspace vault at event time.
2. **Add credentials first** via `zombiectl credential add`, then `zombiectl up`. Either order works — but `zombiectl up` refuses to deploy until every key listed under `credentials:` is present in the vault.

```bash
zombiectl credential add slack_bot_token --value=$SLACK_BOT_TOKEN
zombiectl credential add github_pat --value=$GITHUB_PAT
zombiectl up
```

See [Starting and observing](/zombies/running) for the deployment workflow.
