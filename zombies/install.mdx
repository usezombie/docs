---
title: "Installing a zombie"
description: "What `zombiectl install --from <path>` does and how to invoke it."
---

`zombiectl install --from <path>` is how you bring a zombie up. It reads `SKILL.md` and `TRIGGER.md` from the directory you point at, validates them, and uploads to your active workspace. **Install is the deploy** — there's no separate `up` step.

```bash
zombiectl install --from ~/.config/usezombie/samples/platform-ops
```

```
✓ platform-ops-zombie installed.
  zombie_id:      zmb_2041
  webhook_url:    https://api.usezombie.com/v1/webhooks/zmb_2041
  webhook_secret: (printed once — paste into your GitHub repo webhook config)
```

The webhook secret is shown **once**. Lost it? See [Quickstart → Wire the GitHub webhook](/quickstart#wire-the-github-webhook) for the rotation procedure.

Re-running against the same workspace is safe — running it twice has the same effect as running it once. Your edits to `SKILL.md` / `TRIGGER.md` replace the previous version on the next event.

## Two ways to invoke

Most users don't run `zombiectl install --from` directly. They drive the install from a host agent (Claude Code, Amp, Codex CLI, OpenCode):

```
/usezombie-install-platform-ops
```

The slash command asks three gating questions (Slack channel, production branch glob, optional cron schedule) and prints the webhook URL + secret at the end. See [Quickstart](/quickstart) for the full walkthrough.

Power users and scripted setups call `zombiectl install --from <path>` directly.

## Exit codes

| Code | Meaning |
|---|---|
| `0` | Installed (or updated). |
| `1` | No workspace selected, or API error. |
| `2` | Missing `--from`, invalid path, or schema rejection. |
