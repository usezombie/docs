Memory is what a zombie *learned* from prior events that lets it behave like a teammate who's been here before, instead of one that showed up this morning. A zombie writes facts via `memory_store` and reads them back via `memory_recall` on the next event.

Memory is enabled by default for new zombies.

## The four categories

| Category | Scope | Lifetime | Use for |
|---|---|---|---|
| `core` | zombie | permanent until forgotten | Learned facts about durable entities — customers, accounts, preferences. |
| `daily` | zombie | 72h auto-prune | Time-bounded follow-ups, open tickets, scheduled reminders. |
| `conversation` | event | dies with the stage | Within-stage scratch — checkpoint findings during a long incident. |
| `workspace` | workspace | durable | Facts shared across every zombie in the workspace. |

Pick `core` for "would I want a future event to know this?" Pick `daily` for "this should expire on its own." Pick `conversation` only for staging during a long stage. Pick `workspace` only when multiple zombies need the same fact.

## The four tools

The agent calls these from inside a stage; the runtime wires them to the memory backend.

| Tool | Shape | Purpose |
|---|---|---|
| `memory_store(key, category, content, tags?)` | upsert | Write a distilled fact. |
| `memory_recall(key)` | one entry or empty | Exact lookup by key. |
| `memory_list(filter)` | matching entries | Filter by category / tags / recency. |
| `memory_forget(key)` | delete | Remove or correct a fact. |

There is no vector search. Pre-filter with tags + categories + recency, then let the model reason over the candidate set.

## Where memory fits in a stage

The two memory touchpoints are at the **start** of a stage (`memory_recall` before deciding) and at the **end** (`memory_store` after deciding what was learned). Anything writing to memory in the middle is probably scratch — that belongs in `conversation`.

```mermaid
flowchart LR
    Start([stage opens]) --> Recall["memory_recall<br/>pull prior context"]
    Recall --> Reason[agent reasons + acts]
    Reason --> Store["memory_store<br/>write what was learned"]
    Store --> End([stage closes])
```

## Entry format

Memory is stored in Postgres but exports as markdown for human review.

```markdown
---
key: customer_bob_chen_acme
category: core
tags: [customer, plan_pro]
---

Bob Chen, Acme. Pro since Jan 15. API user (not dashboard), webhooks → Slack.
Prefers technical responses. Prior: Feb 3 signature issue (NTP).
```

Frontmatter is structured metadata; the body is the distilled prose the agent wrote.

## Edit-then-replay

Export memory to markdown, fix or delete entries, import it back:

```bash
zombiectl memory export --zombie {zombie_id} --out ./zombie-memory/
vim ./zombie-memory/core/some_entity.md
zombiectl memory import --zombie {zombie_id} --from ./zombie-memory/
```

The next event sees the corrections.

## Isolation

Every memory operation is scoped by `zombie_id`. Zombie A cannot read Zombie B's memory. The memory store is a separate Postgres database, not a shared filesystem — even an agent with shell or file tools can't reach it through the wrong protocol.

## Common questions

**My zombie keeps re-asking the user for the same info.**

Either `memory_store` isn't being called at the end of the previous event, or `memory_recall` isn't being called at the start of the next one. Audit your `SKILL.md` — recall must happen *before* the reasoning step.

**My zombie remembers something stale.**

The underlying system changed but memory didn't. Export → correct → import, or call `memory_forget(key)` directly. For systems-of-record that change frequently, add a sync job.

**Should I use `core` or `conversation` for this?**

`conversation` only for stage-scoped scratch (e.g. checkpointing tool-call findings during a long incident — see [How long does a zombie run?](/concepts/context-lifecycle)). Everything else is `core` or `daily`.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Zombie "forgets" between events | Memory not enabled | `zombiectl zombie memory enable` |
| Recall returns nothing on a known entity | Inconsistent key convention | Stable key derivation in `SKILL.md`. |
| Memory grows forever | No pruning on `core` (by design) | `zombiectl memory forget` or export → prune → import. |
| `UZ-MEM-001` on store | Memory backend unreachable | `zombiectl zombie memory status`; verify DB connection. |
