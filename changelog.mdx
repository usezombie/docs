<Update label="v0.19.0 ‚Äî Apr 18, 2026" tags={["What's new", "API", "Security"]}>
  ## Unified webhook authentication

  Webhook authentication now runs through a single middleware before your handler sees the request. URL-embedded secrets, Bearer tokens, and HMAC signatures are all verified in one place, with constant-time comparisons throughout and a fail-closed default.

  ## What's new

  - **Config-driven HMAC providers.** Adding a new HMAC-SHA256 provider (Jira, Asana, Stripe, or any future integration) no longer requires a code change ‚Äî declare the signature scheme in your `TRIGGER.md` frontmatter:

    ```yaml
    webhook:
      signature:
        header: "x-jira-hook-signature"
        prefix: "sha256="
        ts_header: "x-jira-timestamp"  # optional
    ```

  - **Built-in provider registry** covers Slack, GitHub, and Linear. Invariants (unique signature headers, non-empty header names) are enforced at build time.
  - **Slack signing secret** is now loaded once at server boot instead of being re-read from the environment on every request.

  ## API reference

  - Error codes are preserved: `UZ-AUTH-001` (URL secret or Bearer mismatch), `UZ-WH-010` (invalid HMAC signature), `UZ-WH-011` (stale timestamp, older than 5 minutes).
  - Failure behavior is fail-closed across every strategy.
</Update>

<Update label="v0.18.0 ‚Äî Apr 16, 2026" tags={["What's new", "API", "UI"]}>
  ## Operator dashboard foundation

  The backbone of the operator dashboard is live: a workspace-wide activity feed, an operator kill switch for runaway zombies, aligned billing summaries for workspaces and individual zombies, and a reusable set of accessible React primitives the dashboard pages will build on.

  ## What's new

  - **Six shared React primitives** with colocated unit tests, ARIA support, and mobile-responsive defaults: `StatusCard`, `EmptyState`, `Pagination` (cursor and page variants), `DataTable`, `ConfirmDialog`, `ActivityFeed`.
  - **Three-layer design tokens** ‚Äî primitive values ‚Üí semantic names (`--primary`, `--success`, `--warning`, `--info`, `--destructive`) ‚Üí Tailwind v4 utility classes (`bg-success`, `text-warning`, `border-info`). Unlocks `bunx shadcn@latest add <primitive>` for future upstream fixes.

  ## API reference

  - `GET /v1/workspaces/{workspace_id}/activity?limit=&cursor=` ‚Äî workspace-wide activity feed across every zombie, cursor-paginated newest-first. Powers the dashboard's Recent Activity panel.
  - `POST /v1/workspaces/{workspace_id}/zombies/{zombie_id}:stop` ‚Äî operator kill switch. Transitions `active` or `paused` ‚Üí `stopped`, emits a `zombie_stopped` activity event. Returns `409 UZ-ZMB-010` on a second call, `404 UZ-ZMB-009` on cross-workspace access.
  - `GET /v1/workspaces/{workspace_id}/zombies/{zombie_id}/billing/summary?period_days=7|30` ‚Äî per-zombie slice of the workspace billing summary. Shares the exact response envelope with the workspace-scoped endpoint.
  - `GET /v1/workspaces/{workspace_id}/billing/summary` now accepts both the legacy `?period=7d|30d|90d` and the new `?period_days=7|30|90`. Response adds a `total_cents` field carrying the sum of billable credits.

  ## Bug fixes

  - `GET /v1/workspaces/{workspace_id}/billing/summary` had been returning zeros since v0.10 (when `usage_ledger` was dropped). It now aggregates real data from execution telemetry through a shared summary store, so workspace and per-zombie summaries can no longer drift.
</Update>

<Update label="v0.17.0 ‚Äî Apr 16, 2026" tags={["Breaking", "API", "CLI"]}>
  ## REST conventions cleanup + full OpenAPI coverage

  List endpoints now share one pagination envelope across every resource, memory reads moved to `GET`, and `openapi.json` covers every live route the server exposes.

  ## Upgrading

  - **List response shape changed** to `{ items, total }` (plus `cursor` where paginated). The per-resource keys ‚Äî `zombies`, `agents`, `grants`, `events` ‚Äî are gone.
  - **Memory read endpoints moved from `POST` to `GET`:**
    - `GET /v1/memory/recall?zombie_id=&query=&limit=` (was `POST /v1/memory/recall`)
    - `GET /v1/memory/list?zombie_id=&category=&limit=` (was `POST /v1/memory/list`)
    - Response field rename: `entries`/`count` ‚Üí `items`/`total`.
    - `memory/store` and `memory/forget` are unchanged.
  - **Upgrade `zombiectl` and the server together.** The CLI has already been updated; rolling back the CLI against the new server will break.

  ## What's new

  - SDK generators can emit a single `Paginated<T>` type across zombies, external agents, integration grants, activity, memory, and telemetry.
  - `DELETE` on external agents and integration grants returns a proper `204 No Content` through the standard envelope instead of a hand-rolled status write.

  ## API reference

  - `public/openapi.json` now documents every route the server exposes. 8 stale paths were removed (`/v1/runs/*` retired in v0.10 and an orphaned `skills/*/secrets/*`), and 26 previously undocumented operations were authored: workspace-scoped zombies, workspace credentials, memory, external agents, integration grants, execute, slack, approval and grant-approval webhooks, internal telemetry, spec relay, and scoring config.
</Update>

<Update label="v0.16.0 ‚Äî Apr 16, 2026" tags={["Breaking", "API", "CLI"]}>
  ## Workspace-scoped REST paths

  Flat `/v1/zombies/*` endpoints have been replaced with workspace-scoped REST paths. Identity ‚Äî workspace, zombie, grant ‚Äî is now always in the URL path, and query parameters are reserved for pagination (`page`, `limit`, `cursor`) and search (`search`).

  ## Upgrading

  - **Flat `/v1/zombies/*` paths return bare `404`.** There are no `410 Gone` stubs (per the pre-v2.0 API teardown policy). Update clients before upgrading the server.
  - **`workspace_id` is no longer accepted in `POST` request bodies** for `zombies` and `credentials` ‚Äî the URL path is authoritative.
  - **`zombiectl` has already been updated.** Upgrade CLI and server together; rolling back the CLI against the new API will break.

  **Route migration:**

  | Before | After |
  | --- | --- |
  | `GET    /v1/zombies/?workspace_id=X` | `GET    /v1/workspaces/{ws}/zombies` |
  | `POST   /v1/zombies/` | `POST   /v1/workspaces/{ws}/zombies` |
  | `DELETE /v1/zombies/{id}` | `DELETE /v1/workspaces/{ws}/zombies/{id}` |
  | `GET    /v1/zombies/activity?zombie_id=X` | `GET    /v1/workspaces/{ws}/zombies/{id}/activity` |
  | `GET    /v1/zombies/credentials?workspace_id=X` | `GET    /v1/workspaces/{ws}/credentials` |
  | `POST   /v1/zombies/credentials` | `POST   /v1/workspaces/{ws}/credentials` |
  | `POST   /v1/zombies/{id}/integration-requests` | `POST   /v1/workspaces/{ws}/zombies/{id}/integration-requests` |
  | `GET    /v1/zombies/{id}/integration-grants` | `GET    /v1/workspaces/{ws}/zombies/{id}/integration-grants` |
  | `DELETE /v1/zombies/{id}/integration-grants/{gid}` | `DELETE /v1/workspaces/{ws}/zombies/{id}/integration-grants/{gid}` |
  | `POST   /v1/zombies/{id}:steer` | `POST   /v1/workspaces/{ws}/zombies/{id}:steer` |

  ## What's new

  - Every workspace-scoped handler authorizes workspace membership after authentication.
  - Cross-workspace zombie lookups return `404`, not `403`, so the API does not leak the existence of resources you cannot see.

  ## API reference

  - `openapi.json` reflects the `:steer` path update. Broader workspace-scoped coverage for CRUD, activity, and grants endpoints arrived in v0.17.0.
</Update>

<Update label="v0.15.0 ‚Äî Apr 15, 2026" tags={["What's new", "API"]}>
  ## Live zombie steering

  Redirect a running zombie mid-execution without killing it. Steer messages are delivered at the next event-loop iteration and injected as synthetic events into the zombie's stream.

  ## What's new

  - Worker polls the steer key at the top of each event loop iteration; messages are queued with a 300-second TTL so they either land mid-execution or wait for the next event.
  - The response body includes `execution_id` so callers can distinguish a mid-execution land from a queued wait.
  - Execution tracking: zombie sessions now record `execution_id` and `execution_started_at` when execution begins, and both clear when execution ends or on worker crash recovery.

  ## API reference

  - `POST /v1/zombies/{id}:steer` ‚Äî writes a message to Redis key `zombie:{id}:steer` with a 300-second TTL.

    Response:

    ```json
    { "message_queued": true, "execution_active": true, "execution_id": "exec_..." }
    ```

  - Error contracts: `401` missing auth ¬∑ `404` cross-workspace zombie ¬∑ `400` empty or >8192-byte message.
</Update>

<Update label="v0.14.0 ‚Äî Apr 12, 2026" tags={["What's new", "Memory", "API"]}>
  ## Persistent zombie memory

  Zombies now remember facts across runs. Memory persists in Postgres even after a workspace is destroyed, so a lead-collector zombie does not re-research every lead and a customer-support zombie does not re-ask customers their plan.

  ## What's new

  - Each zombie's memory is row-scoped by zombie ID ‚Äî two concurrent zombies cannot read each other's entries.
  - The `memory_store`, `memory_recall`, `memory_list`, and `memory_forget` tools route to a dedicated Postgres schema isolated by a least-privilege role.
  - The `conversation` category remains ephemeral (workspace SQLite) as intended.

  ## API reference

  - `POST /v1/memory/store` ‚Äî store or upsert a key-value entry for a zombie.
  - `POST /v1/memory/recall` ‚Äî keyword search across key and content. *(Moved to `GET` in v0.17.0.)*
  - `POST /v1/memory/list` ‚Äî list entries, optionally filtered by category. *(Moved to `GET` in v0.17.0.)*
  - `POST /v1/memory/forget` ‚Äî delete a specific entry by key (idempotent).
  - All endpoints enforce workspace scope: a caller can only access zombies in their own workspace.
</Update>

<Update label="v0.13.0 ‚Äî Apr 12, 2026" tags={["What's new", "Integrations", "API"]}>
  ## Integration grants and credentialed proxy

  Zombies ‚Äî both internal and external LangGraph or CrewAI agents ‚Äî can now call external services through UseZombie's credentialed proxy. The credential never leaves UseZombie; it is injected server-side and stripped from any accidental echo in the response.

  ## What's new

  - **Integration grants.** A zombie requests a grant for a service; UseZombie fans out an Approve/Deny notification to Slack, Discord, and the dashboard simultaneously. One human click creates a durable grant ‚Äî no per-call approval needed.
  - **Per-request approval gate** still runs independently for high-risk endpoints.
  - **Revoke at any time** via the API or `zombiectl grant revoke`.
  - **External agent keys** (prefix `zmb_`) are created workspace-scoped under Clerk auth; the raw key is shown once and only a SHA-256 hash is stored. Each key resolves to a full zombie identity.
  - **Services supported:** Slack, Gmail / AgentMail, Discord, Grafana.

  ## API reference

  - `POST /v1/zombies/{id}/integration-requests` ‚Äî request a grant for a service.
  - `DELETE /v1/zombies/{id}/integration-grants/{grant_id}` ‚Äî revoke a grant.
  - `POST /v1/execute` ‚Äî credentialed proxy call. Pipeline: grant check ‚Üí firewall (domain, endpoint, injection) ‚Üí approval gate ‚Üí credential inject ‚Üí outbound HTTP ‚Üí credential echo strip ‚Üí activity log. Returns the service response plus `X-UseZombie-Action-Id` and `X-UseZombie-Firewall-Decision` headers.
  - Authentication: zombie session (internal) or `Authorization: Bearer zmb_xxx` (external agent).
  - New error codes: `UZ-APIKEY-001`, `UZ-APIKEY-002`, `UZ-GRANT-001`, `UZ-GRANT-002`, `UZ-GRANT-003`, `UZ-PROXY-001`.

  ## CLI

  - `zombiectl agent create / list / delete`
  - `zombiectl grant list / revoke`
</Update>

<Update label="v0.12.0 ‚Äî Apr 12, 2026" tags={["What's new", "Observability", "API"]}>
  ## Zombie execution telemetry

  Every zombie event delivery now records `token_count`, `time_to_first_token_ms`, `wall_seconds`, and `credit_deducted_cents`, with two query surfaces ‚Äî one workspace-scoped, one cross-workspace for operators.

  ## What's new

  - Writes are idempotent on `event_id` ‚Äî crash-recovery redelivery of the same event never creates duplicate rows.
  - Telemetry write failures are logged but never affect credit deduction or event acknowledgement.
  - Each delivery also emits an OpenTelemetry span (`zombie.delivery`) with `zombie_id`, `workspace_id`, `event_id`, `token_count`, and `plan_tier` attributes. The span uses the real wall-clock delivery window, so it appears at the correct position in Grafana Tempo.

  ## API reference

  - `GET /v1/workspaces/{ws}/zombies/{id}/telemetry?limit=50&cursor=` ‚Äî workspace-scoped, cursor-paginated, newest-first.
  - `GET /internal/v1/telemetry?workspace_id=&zombie_id=&after=&limit=100` ‚Äî operator cross-workspace query. All parameters are optional; `after` accepts epoch milliseconds for time-window queries.
</Update>

<Update label="v0.11.0 ‚Äî Apr 12, 2026" tags={["What's new", "Integrations"]}>
  ## Slack plugin

  Workspaces can now connect Slack via the "Add to Slack" OAuth flow or the `zombiectl credential add slack` CLI path. The bot token is stored in the zombie vault; the workspace integrations table holds only routing metadata ‚Äî credentials and routing never commingle.

  ## What's new

  - OAuth state is HMAC-signed and the one-time nonce has a 10-minute replay-protection TTL in Redis.
  - Event and interaction signatures are verified with constant-time HMAC-SHA256.
  - Bot-loop events are filtered; the workspace is resolved via the integrations table and the event is enqueued to the zombie configured with a `slack_event` trigger.

  ## API reference

  - `GET /v1/slack/install` ‚Äî redirects workspace owners to Slack OAuth with an HMAC-signed CSRF state and a 10-minute Redis replay-protection nonce.
  - `GET /v1/slack/callback` ‚Äî exchanges the OAuth code, stores the bot token in the vault, creates the routing record, and posts a one-time confirmation message.
  - `POST /v1/slack/events` ‚Äî signature-verified, bot-loop-filtered event ingestion.
  - `POST /v1/slack/interactions` ‚Äî signature-verified; URL-decodes the Slack `payload=` form field and relays `gate_*` approve/deny actions to the approval gate.
</Update>

<Update label="v0.10.0 ‚Äî Apr 12, 2026" tags={["New releases", "Observability"]}>

<Update label="v0.9.0 ‚Äî Apr 12, 2026" tags={["New releases", "Billing"]}>
  ## Zombie credit metering (M15_001)

  Free-plan zombies now deduct from `consumed_credit_cents` after each successful event
  delivery, at 1 cent per agent-second. Previously, the pre-execution credit gate blocked
  exhausted workspaces but the balance never decremented ‚Äî free-plan zombies effectively
  consumed unlimited quota. Now:

  - **Free plan:** Each delivery records a `CREDIT_DEDUCTED` audit row and updates
    `consumed_credit_cents` / `remaining_credit_cents` atomically.
  - **Scale plan:** Short-circuits with no DB write ‚Äî Scale is unlimited.
  - **Exhausted credit:** Still writes a zero-delta audit row for observability, so
    operators can see post-exhaustion usage events.
  - **Crash recovery:** Replay of the same `event_id` is idempotent ‚Äî no double-charge.
  - **DB failure:** Logged and the event is still acknowledged, so a transient Postgres
    outage never causes message loss or redelivery storms.

  No API changes.
</Update>

<Update label="v0.8.0 ‚Äî April 11, 2026" tags={["New releases", "Zombies", "API"]}>
  ## Zombie directory format, AI Firewall, error standardization, pipeline v1 removal

  ### Zombie directory format
  Zombies are now two-file directories (`SKILL.md` + `TRIGGER.md`) instead of a single `.md` file.
  `SKILL.md` follows the ClaHub registry format ‚Äî the same file you upload to the CLI is publishable to the skill registry.
  `TRIGGER.md` carries deployment config: trigger, chain, budget, network policy, credentials.
  `zombiectl install` scaffolds both files; `zombiectl up` sends them raw to the API.

  ### Dynamic skills (no compiled Zig per skill)
  Skills are now config-driven. The NullCraw executor reads `SKILL.md` instructions and uses
  built-in tools (`shell`, `http`, `file_read`) to call external APIs. Adding a new skill requires
  only a new directory ‚Äî no rebuild of the server binary.

  ### AI Firewall ‚Äî 4-layer outbound inspection
  Every outbound request from a Zombie now passes through an AI Firewall before reaching external APIs:
  - **Domain allowlist** ‚Äî only domains declared in `TRIGGER.md` `network.allow` can be reached
  - **Endpoint policy** ‚Äî per-endpoint rules in `TRIGGER.md` `firewall:` section (e.g., allow GET, deny POST)
  - **Prompt injection detection** ‚Äî scans outbound bodies for instruction override, role hijacking, and jailbreak patterns
  - **Content scanning** ‚Äî inspects response bodies for credential leakage and PII (credit cards, SSNs, API keys)
  All firewall decisions are logged as activity events. Fails closed on errors.

  ### API error format standardized (RFC 7807)
  All error responses now use `application/problem+json` with `UZ-` prefixed error codes.
  Every error code has a stable HTTP status ‚Äî callers no longer need to parse HTTP status codes independently.

  ### Pipeline v1 removed
  The v1 GitHub PR-solver pipeline has been removed. All `/v1/runs/*` and `/v1/specs` endpoints
  return **HTTP 410 Gone** with error code `ERR_PIPELINE_V1_REMOVED`. Use zombie-native SSE stream
  and chat-inject API instead (see v0.5.0 release notes).

  ### Webhook auth ‚Äî URL-embedded secret
  Preferred webhook URL format: `POST /v1/webhooks/{zombie_id}/{secret}`.
  Bearer token remains supported as fallback.

  ### Handler context layer (internal)
  All HTTP handler boilerplate (arena setup, request ID, Bearer auth) is now handled by a shared
  `hx.zig` wrapper. Handlers contain only business logic. No user-visible behavior change.
</Update>

<Update label="v0.5.0 ‚Äî April 11, 2026" tags={["New releases", "Zombies"]}>
  ## Lead Zombie ‚Äî v2 core ships

  UseZombie is now a runtime for always-on agents. Two commands, running agent:

  ```bash
  zombiectl install lead-collector
  zombiectl up
  ```

  ## What's new

  ### Zombie config format
  YAML frontmatter (trigger, skills, credentials, budget) + markdown body (agent instructions).
  The CLI compiles YAML ‚Üí JSON before upload; the server only ever sees JSON.
  Supports voice-transcribed instructions as the instruction body.

  ### Webhook ingestion
  Every zombie gets a stable inbound URL: `POST /v1/webhooks/{zombie_id}`.
  Routing is by primary key ‚Äî no source name collisions, no JSONB index.
  Bearer token auth per zombie. Idempotency via Redis SET NX (24h TTL).
  Returns 202 Accepted or 200 Duplicate.

  ### Activity stream
  Append-only audit log (`core.activity_events`). Every zombie action ‚Äî event received,
  skill invoked, response returned ‚Äî is timestamped and queryable.
  `zombiectl logs` streams the activity log. Cursor-based pagination for replay.

  ### Credential injection
  Credentials are resolved from the vault at runtime and injected into the sandbox.
  No credentials in config files. Add credentials with `zombiectl credential add`.

  ### Session checkpoint
  The zombie's conversation context is checkpointed to Postgres after each event.
  On crash and restart, the zombie resumes from the last checkpoint ‚Äî no lost context.

  ### New CLI commands
  `zombiectl install`, `zombiectl up`, `zombiectl status`, `zombiectl kill`,
  `zombiectl logs`, `zombiectl credential add`, `zombiectl credential list`.

  ### Schema additions
  - `core.zombies` ‚Äî zombie registry with JSONB config
  - `core.zombie_sessions` ‚Äî session checkpoint (context upserted after each event)
  - `core.activity_events` ‚Äî append-only audit log (UPDATE/DELETE blocked by trigger)

  Applied automatically by `zombied migrate`. No changes to existing tables.

  ### API reference updated
  16 v1 endpoints removed from the OpenAPI spec (agents, harness, specs endpoints no longer in v2 path).
  `POST /v1/webhooks/{zombie_id}` added. Mintlify sync required ‚Äî see [API Reference](/api-reference/introduction).

  ### Version tooling
  `make sync-version` / `make check-version` prevent VERSION drift across `build.zig.zon` and `zombiectl/package.json`.

  ### Bug fixes
  - Fixed YAML parser silently dropping array items in CLI config upload
  - Fixed UTF-8 truncation splitting multi-byte characters in session context
</Update>

<Update label="v0.4.0 ‚Äî April 6, 2026" tags={["New releases", "Improvements"]}>
  ## Steer running agents mid-run

  Interrupt a running agent without aborting it. Send a message via `zombiectl runs interrupt <run_id> <message>` or `POST /v1/runs/{id}:interrupt` ‚Äî the agent picks it up at the next gate checkpoint. Two modes: **queued** (next checkpoint) and **instant** (IPC delivery).

  ## Live run streaming (CLI)

  `zombiectl run --spec <file> --watch` now streams gate results in real time. Reconnect with `Last-Event-ID` replays only missed events ‚Äî no duplicate floods. Ctrl+C works cleanly.

  ## Run replay (CLI)

  `zombiectl runs replay <run_id>` prints a per-gate narrative for completed runs ‚Äî exit codes, stdout/stderr, wall time, step by step.

  ## Workspace billing breakdown

  `zombiectl workspace billing --workspace-id <id>` shows completed, non-billable, and score-gated runs with optional `--period` and `--json` flags. Backed by `GET /v1/workspaces/{id}/billing/summary`.

  ## Agent run observability

  Every run now produces a full trace tree in Grafana Tempo ‚Äî query `{run.id="<id>"}` for a waterfall of agent calls and gate checks. Per-workspace Prometheus metrics: token consumption, run outcomes, and gate repair loop distribution.

  ## Resource efficiency scoring

  Agent runs are now scored on actual memory and CPU usage. Agents that stay within their resource limits score higher. Score formula updated to v2 with real resource data.

  ## Breaking change

  SSE `id:` field on live events changed from sequential counter to `created_at` Unix milliseconds. Clients parsing `Last-Event-ID` as a sequence number must update.
</Update>
