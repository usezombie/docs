<Update label="v0.22.0 — Apr 19, 2026" tags={["Breaking", "API", "CLI"]}>
  ## Legacy spec/run commands and endpoints removed

  The v1 spec→run flow is gone. Zombies are the product model now — long-lived agents you install, steer, and kill — and the old "submit a spec YAML, wait for a run" surface has been removed from both `zombied` and `zombiectl`. Leaving it around was confusing the CLI help, the API surface, and reviewers.

  ## Upgrading

  The following commands and endpoints no longer exist. Migrate to the zombie-centric equivalents (see the Zombies section of the docs) before upgrading:

  - **`zombied` subcommand removed:**
    - `zombied run <spec>` (including `--watch`). The pre-flight `spec-validate` helper is gone with it.
  - **`zombiectl` commands removed:**
    - `zombiectl run` (queue a run) and `zombiectl run status <run_id>`
    - `zombiectl run --spec <file> --preview` / `--preview-only`
    - `zombiectl run --watch`
    - `zombiectl runs list`, `zombiectl runs cancel`, `zombiectl runs replay`, `zombiectl runs interrupt`
    - `zombiectl spec init`
  - **HTTP endpoints removed (return 404):**
    - `POST /v1/workspaces/{workspace_id}/spec/template`
    - `POST /v1/workspaces/{workspace_id}/spec/preview`

  Invoking any removed CLI command now exits non-zero with `unknown command: …`. Clients that still POST to the removed endpoints receive 404 with the standard `application/problem+json` body — no 410 gone stub is provided on pre-v2.0 releases (consistent with the v0.16.0 teardown policy). CLI and server can upgrade independently; both sides of the v1 flow are removed in this release.

  ## API reference

  - Removed `POST /v1/workspaces/{workspace_id}/spec/template` — spec generation via LLM relay.
  - Removed `POST /v1/workspaces/{workspace_id}/spec/preview` — blast-radius preview via LLM relay.
  - All other workspace routes, including `/credentials/llm`, `/zombies`, `/activity`, agent-keys, and tenant API keys, are unchanged.

  ## CLI

  `zombiectl --help` now lists only the commands that map to the zombie-centric model: `login`, `logout`, `workspace`, `specs sync`, `doctor`, `skill-secret`, `admin`, plus the `zombie install|up|status|kill|logs|credential` family. The help banner is five lines shorter.
</Update>

<Update label="v0.20.0 — Apr 18, 2026" tags={["What's new", "API", "Security"]}>
  ## Tenant API keys — multi-key, rotatable, self-service admin auth

  Admin auth is no longer tied to a single process-wide env var. Tenant admins now mint named API keys via the API, each scoped to their tenant, revocable on demand, and identifiable in audit logs. The legacy `API_KEY` environment variable still works as a bootstrap fallback, but it is no longer the primary admin auth path — new installs and production tenants should migrate to tenant API keys.

  Keys are `zmb_t_`-prefixed (6-char prefix + 64 hex chars, 70 total) and are **returned exactly once on creation** — only the SHA-256 hex digest is persisted. The raw key cannot be retrieved again; if lost, revoke it and mint a new one.

  ## What's new

  - **Mint, list, revoke, and delete tenant keys** through the new `/v1/api-keys` endpoint family. Every query is tenant-scoped; no CRUD call can read or mutate another tenant's keys.
  - **Per-request principal carries `user_id` and `tenant_id`** when authenticated with a tenant key — enabling per-tenant isolation, per-key revocation, and audit identity without server restarts.
  - **Bootstrap warning on startup.** When the server boots with only the legacy `API_KEY` env var set, a warning is emitted to nudge operators toward tenant keys (no behavior change; fallback still works).
  - **Workspace-scoped "external agent" keys renamed to "agent keys"** for consistency. The underlying table is now `core.agent_keys`; the public endpoints moved from `/v1/workspaces/{ws}/external-agents` to `/v1/workspaces/{ws}/agent-keys`. Operation IDs renamed from `list_external_agents` / `create_external_agent` / `delete_external_agent` to `list_agent_keys` / `create_agent_key` / `delete_agent_key`. Agent keys (per-zombie) and API keys (per-tenant) are now clearly distinct surfaces.

  ## API reference

  - `POST /v1/api-keys` — mint a new tenant API key. Body: `{ "key_name": "ci-pipeline", "description": "GH Actions deploys" }`. Response: `{ id, key_name, key: "zmb_t_…", created_at }`. Raw `key` shown once only.
  - `GET /v1/api-keys` — paginated list (`page`, `page_size` ≤ 100, `sort` in `created_at`/`-created_at`/`key_name`/`-key_name`). Returns metadata only; `last_used_at` is always `null` in this release and will populate in a future async-stamping update.
  - `PATCH /v1/api-keys/{id}` — revoke a key. Body must be exactly `{ "active": false }`; re-activation is not supported (mint a new key instead).
  - `DELETE /v1/api-keys/{id}` — hard-delete a revoked key. Returns `409 UZ-APIKEY-008` if the key is still active — revoke first.
  - New error codes: `UZ-APIKEY-003` (not found), `UZ-APIKEY-004` (revoked, auth path), `UZ-APIKEY-005` (name taken — 409), `UZ-APIKEY-006` (already revoked), `UZ-APIKEY-007` (attempt to set `active: true`), `UZ-APIKEY-008` (active key must be revoked before deletion).
  - Public OpenAPI (`public/openapi.json`) adds the `API Keys` tag, schema `TenantApiKey`, and the four paths above. The `External Agents` tag and `ExternalAgent` schema were renamed to `Agent Keys` / `AgentKey` with no functional change to workspace-agent behavior.
</Update>

<Update label="v0.19.0 — Apr 18, 2026" tags={["Breaking", "What's new", "API", "CLI", "Security", "Integrations"]}>
  ## Unified webhook authentication — seven first-class providers

  Every per-zombie webhook now flows through a single authentication middleware before your handler sees the request. URL-embedded secrets, Bearer tokens, HMAC signatures, and Svix multi-signature rotation are all verified in one place, with constant-time comparisons throughout and a fail-closed default.

  Seven first-class providers ship in this release: **agentmail**, **Grafana**, **Slack**, **GitHub**, **Linear**, **Jira**, and **Clerk** (via Svix). Each authenticates through the middleware that fits its model — no per-provider branches in handler code, and no env reads on the hot path.

  ## Upgrading

  - **Custom-method endpoints (`:action`) moved to subpath form (`/action`).** Google-style `:action` URL syntax (e.g. `/v1/workspaces/{ws}/zombies/{id}:steer`) has been replaced with a direct subpath (`/v1/workspaces/{ws}/zombies/{id}/steer`) so the public OpenAPI spec validates cleanly and standard documentation tooling can parameterize it. Affected endpoints: `:steer`, `:stop`, `:approval`, `:grant-approval`, `:pause`, `:sync`. **Upgrade `zombiectl` and the server together** — rolling back the CLI against the new server will break. If you construct these URLs directly (e.g. from a script or SDK), update them to the new form.

    | Before | After |
    | --- | --- |
    | `POST /v1/webhooks/{zombie_id}:approval` | `POST /v1/webhooks/{zombie_id}/approval` |
    | `POST /v1/webhooks/{zombie_id}:grant-approval` | `POST /v1/webhooks/{zombie_id}/grant-approval` |
    | `POST /v1/workspaces/{workspace_id}/zombies/{zombie_id}:steer` | `POST /v1/workspaces/{workspace_id}/zombies/{zombie_id}/steer` |
    | `POST /v1/workspaces/{workspace_id}/zombies/{zombie_id}:stop` | `POST /v1/workspaces/{workspace_id}/zombies/{zombie_id}/stop` |
    | `POST /v1/workspaces/{workspace_id}:pause` | `POST /v1/workspaces/{workspace_id}/pause` |
    | `POST /v1/workspaces/{workspace_id}:sync` | `POST /v1/workspaces/{workspace_id}/sync` |

    Reserved path segments: the webhook action endpoints (`approval`, `grant-approval`) and the Svix route prefix (`svix`) now occupy the URL-secret slot on `/v1/webhooks/{zombie_id}/...`. Webhook URL secrets cannot use those literal values.

  ## What's new

  - **One-line onboarding for registry providers.** GitHub, Linear, and Clerk webhooks each require one field in your `TRIGGER.md` — the middleware picks up the correct signature scheme automatically:

    ```yaml
    # GitHub — one field
    trigger:
      type: webhook
      source: "github"
      signature:
        secret_ref: "github_webhook_secret"

    # Linear — identical shape
    trigger:
      type: webhook
      source: "linear"
      signature:
        secret_ref: "linear_webhook_secret"

    # Clerk — one field; route path /v1/webhooks/svix/{zombie_id} selects the Svix middleware
    trigger:
      type: webhook
      source: "clerk"
      signature:
        secret_ref: "clerk_webhook_secret"
    ```

  - **Custom HMAC providers stay configuration-only.** Providers outside the registry (Jira today, Asana and Stripe tomorrow) declare the scheme explicitly in `TRIGGER.md` — no code change required:

    ```yaml
    trigger:
      type: webhook
      source: "jira"
      signature:
        secret_ref: "jira_webhook_secret"
        header: "x-jira-hook-signature"
        prefix: "sha256="
    ```

  - **Clerk via Svix is supported end-to-end.** A dedicated middleware handles Svix v1 multi-signature HMAC, including key rotation: the middleware iterates every space-delimited entry in the `svix-signature` header and accepts the first constant-time match. Timestamp freshness (5-minute window) and tampered-body detection are enforced automatically.
  - **Per-zombie webhook secrets are vault-backed.** You declare a `secret_ref`; the secret itself is stored encrypted per workspace and loaded only at verification time. Rotating a secret is a vault update — no zombie reconfigure.

  ## API reference

  - **New route:** `POST /v1/webhooks/svix/{zombie_id}` — entry point for Clerk (and any future Svix-signed sender). Returns `202` on valid signature; `401 UZ-WH-010` on tamper/bad signature; `401 UZ-WH-011` on stale `svix-timestamp`.
  - **Existing route:** `POST /v1/webhooks/{zombie_id}` continues to serve agentmail, Grafana, GitHub, Linear, and Jira — the middleware dispatches to URL-secret, Bearer, or HMAC verification based on what the zombie declares.
  - **Error codes** are preserved and now apply consistently across every provider: `UZ-AUTH-002` (URL secret or Bearer mismatch), `UZ-WH-010` (invalid HMAC / Svix signature), `UZ-WH-011` (stale timestamp, older than 5 minutes).
  - **Fail-closed** is the default across every strategy. An unreachable vault or missing secret returns 401 — the handler never sees an unauthenticated request.
</Update>

<Update label="v0.15.0 — Apr 15, 2026" tags={["What's new", "API"]}>
  ## Live zombie steering

  Redirect a running zombie mid-execution without killing it. Steer messages are delivered at the next event-loop iteration and injected as synthetic events into the zombie's stream.

  ## What's new

  - Worker polls the steer key at the top of each event loop iteration; messages are queued with a 300-second TTL so they either land mid-execution or wait for the next event.
  - The response body includes `execution_id` so callers can distinguish a mid-execution land from a queued wait.
  - Execution tracking: zombie sessions now record `execution_id` and `execution_started_at` when execution begins, and both clear when execution ends or on worker crash recovery.

  ## API reference

  - `POST /v1/zombies/{id}:steer` — writes a message to Redis key `zombie:{id}:steer` with a 300-second TTL. *(Moved to workspace-scoped path and renamed to `/steer` in later releases — see v0.16.0 and v0.19.0 Upgrading.)*

    Response:

    ```json
    { "message_queued": true, "execution_active": true, "execution_id": "exec_..." }
    ```

  - Error contracts: `401` missing auth · `404` cross-workspace zombie · `400` empty or >8192-byte message.
</Update>

<Update label="v0.14.0 — Apr 12, 2026" tags={["What's new", "Memory", "API"]}>
  ## Persistent zombie memory

  Zombies now remember facts across runs. Memory persists in Postgres even after a workspace is destroyed, so a lead-collector zombie does not re-research every lead and a customer-support zombie does not re-ask customers their plan.

  ## What's new

  - Each zombie's memory is row-scoped by zombie ID — two concurrent zombies cannot read each other's entries.
  - The `memory_store`, `memory_recall`, `memory_list`, and `memory_forget` tools route to a dedicated Postgres schema isolated by a least-privilege role.
  - The `conversation` category remains ephemeral (workspace SQLite) as intended.

  ## API reference

  - `POST /v1/memory/store` — store or upsert a key-value entry for a zombie.
  - `POST /v1/memory/recall` — keyword search across key and content. *(Moved to `GET` in v0.17.0.)*
  - `POST /v1/memory/list` — list entries, optionally filtered by category. *(Moved to `GET` in v0.17.0.)*
  - `POST /v1/memory/forget` — delete a specific entry by key (idempotent).
  - All endpoints enforce workspace scope: a caller can only access zombies in their own workspace.
</Update>

<Update label="v0.13.0 — Apr 12, 2026" tags={["What's new", "Integrations", "API"]}>
  ## Integration grants and credentialed proxy

  Zombies — both internal and external LangGraph or CrewAI agents — can now call external services through UseZombie's credentialed proxy. The credential never leaves UseZombie; it is injected server-side and stripped from any accidental echo in the response.

  ## What's new

  - **Integration grants.** A zombie requests a grant for a service; UseZombie fans out an Approve/Deny notification to Slack, Discord, and the dashboard simultaneously. One human click creates a durable grant — no per-call approval needed.
  - **Per-request approval gate** still runs independently for high-risk endpoints.
  - **Revoke at any time** via the API or `zombiectl grant revoke`.
  - **External agent keys** (prefix `zmb_`) are created workspace-scoped under Clerk auth; the raw key is shown once and only a SHA-256 hash is stored. Each key resolves to a full zombie identity.
  - **Services supported:** Slack, Gmail / AgentMail, Discord, Grafana.

  ## API reference

  - `POST /v1/zombies/{id}/integration-requests` — request a grant for a service.
  - `DELETE /v1/zombies/{id}/integration-grants/{grant_id}` — revoke a grant.
  - `POST /v1/execute` — credentialed proxy call. Pipeline: grant check → firewall (domain, endpoint, injection) → approval gate → credential inject → outbound HTTP → credential echo strip → activity log. Returns the service response plus `X-UseZombie-Action-Id` and `X-UseZombie-Firewall-Decision` headers.
  - Authentication: zombie session (internal) or `Authorization: Bearer zmb_xxx` (external agent).
  - New error codes: `UZ-APIKEY-001`, `UZ-APIKEY-002`, `UZ-GRANT-001`, `UZ-GRANT-002`, `UZ-GRANT-003`, `UZ-PROXY-001`.

  ## CLI

  - `zombiectl agent create / list / delete`
  - `zombiectl grant list / revoke`
</Update>

  ## Zombie execution telemetry

  Every zombie event delivery now records `token_count`, `time_to_first_token_ms`, `wall_seconds`, and `credit_deducted_cents`, with two query surfaces — one workspace-scoped, one cross-workspace for operators.

  - `GET /internal/v1/telemetry?workspace_id=&zombie_id=&after=&limit=100` — operator cross-workspace query. All parameters are optional; `after` accepts epoch milliseconds for time-window queries.

<Update label="v0.11.0 — Apr 12, 2026" tags={["What's new", "Integrations"]}>
  ## Slack plugin

  Workspaces can now connect Slack via the "Add to Slack" OAuth flow or the `zombiectl credential add slack` CLI path. The bot token is stored in the zombie vault; the workspace integrations table holds only routing metadata — credentials and routing never commingle.

  ## What's new

  - OAuth state is HMAC-signed and the one-time nonce has a 10-minute replay-protection TTL in Redis.
  - Event and interaction signatures are verified with constant-time HMAC-SHA256.
  - Bot-loop events are filtered; the workspace is resolved via the integrations table and the event is enqueued to the zombie configured with a `slack_event` trigger.

  ## API reference

  - `GET /v1/slack/install` — redirects workspace owners to Slack OAuth with an HMAC-signed CSRF state and a 10-minute Redis replay-protection nonce.
  - `GET /v1/slack/callback` — exchanges the OAuth code, stores the bot token in the vault, creates the routing record, and posts a one-time confirmation message.
  - `POST /v1/slack/events` — signature-verified, bot-loop-filtered event ingestion.

  ## Zombie credit metering (M15_001)

  Free-plan zombies now deduct from `consumed_credit_cents` after each successful event
  delivery, at 1 cent per agent-second. Previously, the pre-execution credit gate blocked
  exhausted workspaces but the balance never decremented — free-plan zombies effectively
  consumed unlimited quota. Now:

  - **Free plan:** Each delivery records a `CREDIT_DEDUCTED` audit row and updates
    `consumed_credit_cents` / `remaining_credit_cents` atomically.
  - **Scale plan:** Short-circuits with no DB write — Scale is unlimited.
  - **Exhausted credit:** Still writes a zero-delta audit row for observability, so
    operators can see post-exhaustion usage events.

    outage never causes message loss or redelivery storms.

<Update label="v0.8.0 — April 11, 2026" tags={["New releases", "Zombies", "API"]}>
  ## Zombie directory format, AI Firewall, error standardization, pipeline v1 removal

  ### Zombie directory format
  Zombies are now two-file directories (`SKILL.md` + `TRIGGER.md`) instead of a single `.md` file.
  `SKILL.md` follows the ClaHub registry format — the same file you upload to the CLI is publishable to the skill registry.
  `TRIGGER.md` carries deployment config: trigger, chain, budget, network policy, credentials.
  `zombiectl install` scaffolds both files; `zombiectl up` sends them raw to the API.

  ### Dynamic skills (no compiled Zig per skill)
  Skills are now config-driven. The NullCraw executor reads `SKILL.md` instructions and uses
  built-in tools (`shell`, `http`, `file_read`) to call external APIs. Adding a new skill requires
  only a new directory — no rebuild of the server binary.

  ### AI Firewall — 4-layer outbound inspection
  Every outbound request from a Zombie now passes through an AI Firewall before reaching external APIs:
  - **Domain allowlist** — only domains declared in `TRIGGER.md` `network.allow` can be reached
  - **Endpoint policy** — per-endpoint rules in `TRIGGER.md` `firewall:` section (e.g., allow GET, deny POST)
  - **Prompt injection detection** — scans outbound bodies for instruction override, role hijacking, and jailbreak patterns
  - **Content scanning** — inspects response bodies for credential leakage and PII (credit cards, SSNs, API keys)
  All firewall decisions are logged as activity events. Fails closed on errors.

  ### API error format standardized (RFC 7807)
  All error responses now use `application/problem+json` with `UZ-` prefixed error codes.
  Every error code has a stable HTTP status — callers no longer need to parse HTTP status codes independently.

  ### Pipeline v1 removed
  The v1 GitHub PR-solver pipeline has been removed. All `/v1/runs/*` and `/v1/specs` endpoints
  return **HTTP 410 Gone** with error code `ERR_PIPELINE_V1_REMOVED`. Use zombie-native SSE stream
  and chat-inject API instead (see v0.5.0 release notes).

  ### Webhook auth — URL-embedded secret
  Preferred webhook URL format: `POST /v1/webhooks/{zombie_id}/{secret}`.
  Bearer token remains supported as fallback.

  ### Handler context layer (internal)
  All HTTP handler boilerplate (arena setup, request ID, Bearer auth) is now handled by a shared
  `hx.zig` wrapper. Handlers contain only business logic. No user-visible behavior change.
</Update>

<Update label="v0.4.0 — April 6, 2026" tags={["New releases", "Improvements"]}>
  ## Steer running agents mid-run

  Interrupt a running agent without aborting it. Send a message via `zombiectl runs interrupt <run_id> <message>` or `POST /v1/runs/{id}:interrupt` — the agent picks it up at the next gate checkpoint. Two modes: **queued** (next checkpoint) and **instant** (IPC delivery).

  ## Live run streaming (CLI)

  `zombiectl run --spec <file> --watch` now streams gate results in real time. Reconnect with `Last-Event-ID` replays only missed events — no duplicate floods. Ctrl+C works cleanly.

  ## Run replay (CLI)

  `zombiectl runs replay <run_id>` prints a per-gate narrative for completed runs — exit codes, stdout/stderr, wall time, step by step.

  ## Workspace billing breakdown

  `zombiectl workspace billing --workspace-id <id>` shows completed, non-billable, and score-gated runs with optional `--period` and `--json` flags. Backed by `GET /v1/workspaces/{id}/billing/summary`.

  ## Agent run observability

  Every run now produces a full trace tree in Grafana Tempo — query `{run.id="<id>"}` for a waterfall of agent calls and gate checks. Per-workspace Prometheus metrics: token consumption, run outcomes, and gate repair loop distribution.

  ## Resource efficiency scoring

  Agent runs are now scored on actual memory and CPU usage. Agents that stay within their resource limits score higher. Score formula updated to v2 with real resource data.

  ## Breaking change

  SSE `id:` field on live events changed from sequential counter to `created_at` Unix milliseconds. Clients parsing `Last-Event-ID` as a sequence number must update.
</Update>
